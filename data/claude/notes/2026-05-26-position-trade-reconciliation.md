# Position-Trade Reconciliation · 2026-05-26

## 背景

ECS 上 claude 与 codex 两个账户的持仓数与净成交数看起来对不上：

| 账户 | 5-18 买 | 5-25 买 | 5-25 卖 | 朴素净额 (buy − sell) | positions.csv 实际 |
|------|--------:|--------:|--------:|---------------------:|-------------------:|
| claude | 81 | 63 | 59 | **85** | **93** |
| codex  | 82 | 64 | 72 | **74** | **93** |

claude 差 8 个,codex 差 19 个。怀疑：要么是模拟器记账有 bug,要么是 SELL 里混了"部分减仓"——朴素公式把每笔 SELL 都当成清仓,会低估持仓数。

## 方法

把 `trades.csv` 按时间顺序回放,对每个标的累加 buy 份额、减 sell 份额,得到回放终态;与 `positions.csv` 逐 code 比对(集合差、数量差)。SELL 记账类型按"卖出后剩余 qty"分类：

- `full` (清仓): qty → 0,等价于"该持仓从 portfolio 移除"。
- `partial` (减仓): qty > 0,该持仓继续保留。
- `oversold` (qty < 0): 不应出现,若出现即模拟器 bug。

脚本：`/tmp/reconcile/reconcile.py`(本地一次性脚本,未提交)。

## 结论：模拟器记账正确,无需修复

### 对账总表

| 账户 | 5-18 建仓 | 5-25 新买 | 5-25 卖单 | 其中清仓 | 其中减仓 | 异常(oversold) | 回放终态 | positions.csv | 集合差 / 数量差 |
|------|---------:|---------:|---------:|--------:|--------:|--------------:|--------:|-------------:|----------------:|
| claude | 81 | 63 | 59 | **51** | **8** | 0 | 93 | 93 | 0 / 0 |
| codex  | 82 | 64 | 72 | **53** | **19** | 0 | 93 | 93 | 0 / 0 |

回放与 positions.csv **完全一致**:无"只在回放里"的 code、无"只在实际里"的 code、无数量差异、无 oversold 事件。

### 重写公式

正确的关系是 `期末 = 期初 + 新买 − 清仓` (减仓不计):

- claude: 81 + 63 − 51 = **93** ✓
- codex:  82 + 64 − 53 = **93** ✓

集合上两次买单互不重叠,5-25 的卖单全部从 5-18 的初始组合里出（claude 59/59、codex 72/72 命中初始集合）,也印证了上述公式中"减仓"是减半仓位、"清仓"才让 code 真正退出组合。

### 为何两边都恰好 93（修正）

⚠️ 第一版 note 写的"top_n 合计 = 93"是错的——`configs/competition.yaml` 实际是
`accounts.hs300.top_n = 50` + `accounts.zz500.top_n = 50` = **100**。

两边都落 93 的真实原因是**初始建仓凑不齐**：5-18 第一次建仓时受 `cash` +
`lot_size=100` + 单股价 + `max_single_weight=5%` 联合约束,claude 只填了 81 个、
codex 填了 82 个 (本来应该都 100)。5-25 rebalance 的 net 变化 (claude +4 /
codex -8 之后又被减仓事件抵消) 让两边都收敛到 93。

这是个**巧合数字**,不是结构性的。下一次 rebalance 持仓数还会变。

## 是否要改什么

**不需要。** 模拟器 buy/sell 记账是对的,可以放心信任 `positions.csv` 与 `trades.csv` 的内部一致性。误差源是观察侧的朴素公式,而不是模拟器侧的会计。

如果未来要在 dashboard 或 briefing 里加"周内成交摘要",应同时列出 `full_sells` 与 `partial_sells` 两个数,避免再让人误判。具体增改属于 source 代码改动,本 note 不直接处理,留给周报或后续 ECS 操作。
